## High Severity

### H1. Per-message SQLite query on the gateway hot path

- **Where:** `internal/bot/handler_message.go:25` → `internal/store/queries.go:89`
- **What:** `onMessageCreate` fires for every non-bot message in every guild; each one runs
  `GetChannelByID` through modernc.org/sqlite with `SetMaxOpenConns(1)` (`store.go:34`) and
  no prepared-statement reuse. During a spam raid — the bot's core scenario — hundreds of
  handler goroutines serialize on one connection, contending with the punishment writes
  themselves. The honeypot-channel set is one row per guild and changes only on setup/delete.
- **Flagged by:** 6 of 14 agents independently (database, algorithms, data structures,
  caching, concurrency, security). Unanimous #1 issue.
- **Fix:** Load `AllChannels()` into a `sync.RWMutex`-guarded `map[snowflake.ID]store.Channel`
  at startup; update it in `SetChannel`, `RemoveChannel`, `onChannelDelete`/`onThreadDelete`,
  and `DeleteGuild` paths. A map miss is authoritative (full set is preloaded), so no
  negative-cache or DB fallback is needed.
- **Alternatives:** (a) prepared `*sql.Stmt` hoisted in `Open()` — smaller change, still pays
  the single-connection mutex; (b) allow multiple read connections (WAL supports concurrent
  readers) while keeping one writer — more invasive, weaker win.
- **Impact:** Eliminates ~100% of steady-state DB traffic; per-message cost drops from
  µs-with-contention to a ~ns map lookup. Also enables M2's fix for free.

### H2. DB errors silently conflated with "not a honeypot channel"

- **Where:** `handler_message.go:25-33`; same pattern in `housekeeping.go:8,14,26,32`
- **What:** `if err != nil || hpChannel == nil` treats a SQLite failure (disk full, I/O error,
  residual BUSY) identically to a non-match: no ban, no alert, no log line at any level. A
  failing database silently turns the bot into a no-op. This is the only place in the codebase
  where errors are dropped without a log — everywhere else error discipline is good.
- **Flagged by:** logging, resilience, and architecture agents — all rated it their top gap.
- **Fix:** Split the branches: on `err != nil`, `b.Log.Error(...)` and return; fall through to
  the not-configured path only when the row is genuinely absent. Same for `GetConfig` at
  line 31 and the housekeeping guards. ~4 lines.
- **Impact:** Converts an invisible total-outage mode into an immediately diagnosable one.

### H3. Transient ban failures are permanently dropped

- **Where:** `handler_message.go:94-99` (and softban `DeleteBan` at :103)
- **What:** disgo retries only HTTP 429 (verified against disgo v0.19.6 source — 5xx and
  network errors return immediately). A Discord 5xx or TCP blip during `AddBan` means the
  spammer stays in the server; the dedup key is defer-deleted, so recovery requires them to
  post again. The mod alert itself travels the same possibly-degraded REST path.
- **Fix:** Small retry helper: 2–3 attempts with exponential backoff, retrying only
  network errors / 5xx (`rest.Error` with non-permission codes); keep the existing alert as
  final fallback.
- **Impact:** Converts most transient Discord failures from "spammer not banned" into
  "banned a few seconds late."

### H4. Modal submit risks Discord's 3-second interaction deadline

- **Where:** `handler_honeypot.go:103-167` (`onModalSubmit`)
- **What:** The ephemeral reply is the *initial* interaction callback, sent only after two
  `GetMember` permission checks, an optional `DeleteMessage`, and `ensureWarningMessage`
  (up to `GetMessages(50)` + create + N deletes) — up to ~7 sequential REST round-trips.
  Rate-limit queueing pushes the total past 3s → "This interaction failed" even though the
  config saved.
- **Fix:** `e.DeferCreateMessage(true)` immediately on entry; send results via
  `UpdateInteractionResponse`.
- **Impact:** Eliminates user-visible interaction failures; perceived response becomes instant.

### H5. Redundant `GetMember` REST calls for the bot's own member

- **Where:** `handler_honeypot.go:329-335` (`botPermissionsInChannel`); called twice
  back-to-back at `:135,137`, and twice on guild join (`setup.go:72,119`)
- **What:** Every permission check is a full REST round-trip for the same self-member object.
  The cache config (`bot.go:59`) enables Guilds/Roles/Channels but not Members, so there's no
  cache fallback. Sits directly on H4's latency-critical path.
- **Fix:** Fetch once per handler invocation and pass the `*discord.Member` down; or cache the
  self-member per guild in the existing `cache.TTL` (short TTL — role changes are the only
  invalidator).
- **Impact:** Removes 1–3 REST round-trips (~200–600 ms) per config submit / guild join;
  halves H4's pre-ack latency.

### H6. No backup strategy for the SQLite database

- **Where:** `docker-compose.prod.yml:10-15`
- **What:** All ban/incident history lives in a single named Docker volume on a single server.
  No backup sidecar, no replication, no scheduled dump anywhere in the repo. WAL protects
  against crashes, not host loss / accidental `docker volume rm` / botched migration.
- **Fix:** Litestream sidecar replicating `/data/honeypot.db` to S3-compatible storage
  (designed exactly for WAL-mode SQLite). Minimum: host cron running
  `sqlite3 .backup` and shipping the file off-box.
- **Impact:** Converts unrecoverable data loss into a few-seconds RPO; near-zero runtime cost.

## Medium Severity

### M1. TTL cache never reclaims write-once keys (unbounded `DMs` map)

- **Where:** `internal/cache/ttl.go:30-40`; leaking consumer at `handler_message.go:181-189`
- **What:** Eviction is lazy and per-key on `Get`. The `DMs` cache (24h TTL, keyed by banned
  user ID) is write-once/never-read-again for one-shot spammers, so entries — and Go map
  buckets, which never shrink — accumulate forever. The only unbounded structure in the
  process. `Dedup` is unaffected (explicit `defer Delete`).
- **Fix options:** (a) background sweep ticker (needs a `Stop()`/owner); (b) opportunistic
  sweep on `Set` when `len(m) > N` — keeps the package goroutine-free, ~10 lines;
  (c) max-size bound with eviction on insert.
- **Impact:** Flat memory profile for a daemon that runs for months. Slow-burn issue
  (~50 bytes/spammer), so hygiene rather than urgency.

### M2. Unconditional DB write on every message-delete event

- **Where:** `housekeeping.go:40-44` → `queries.go:140-142`
- **What:** Every deletion of any message in any guild issues
  `UPDATE honeypot_channels SET msg_id = NULL WHERE msg_id = ?` — a WAL write transaction
  that almost always matches 0 rows, on an unindexed column. Bulk purges (provokable via
  other moderation bots) become write storms on the single connection.
- **Fix:** Check the in-memory channel map from H1 first (warning-message IDs are already
  part of `Channel`); only hit the DB for known warning messages. Optionally index `msg_id`.
- **Impact:** ~100% of these writes become a map lookup; purge storms become no-ops.

### M3. Public stats button: uncached `COUNT(*)`, no rate limit

- **Where:** `handler_buttons.go:36-43` → `queries.go:166-169`
- **What:** The counter button sits on the public warning message; unlike the other button
  branches it has no permission check and no rate limit. Each click runs `SELECT COUNT(*)`
  over `honeypot_events` — a table that grows without bound (L2) — through the single
  connection. Click-spammable DB load.
- **Fix options:** (a) cache the count per guild in `cache.TTL` (30–60s); (b) counter column
  on `honeypot_config` incremented in `RecordEvent` (O(1) read, also fixes L2's count cost).
- **Impact:** Bounds worst-case click-spam to one scan per TTL window, or O(1) with the column.

### M4. Config save is two non-atomic store calls; rollback logic leaked to bot layer

- **Where:** `setup.go:57-67`, `handler_honeypot.go:146-155`, `queries.go:62,114`
- **What:** `UpsertConfig` then `SetChannel` (FK requires this order — implicit caller lore).
  `setup.go` hand-rolls compensation via `DeleteGuild`; the modal path has none — a failure
  leaves action/log-channel changed but the honeypot channel not, while telling the admin
  "something went wrong."
- **Fix:** One transactional store method `SaveGuildSetup(cfg Config, channelID snowflake.ID)`;
  both call sites collapse to one call; delete the manual rollback.
- **Impact:** Eliminates half-applied-settings states after mid-setup DB errors.

### M5. No context propagation; unbounded shutdown

- **Where:** all handlers (no `rest.WithCtx` anywhere); `queries.go` uses non-Context
  variants; `main.go:59` calls `b.Close(context.Background())`; `setup.go:84` `time.AfterFunc`
  can fire 150s after shutdown against a closed client
- **What:** On SIGTERM, in-flight handler goroutines (2s DM wait, ban→sleep→unban chains)
  keep issuing uncancellable REST/DB calls; hung Discord calls have no deadline and pin the
  sole DB connection's queue.
- **Fix:** Root context on `Bot` cancelled at shutdown; `rest.WithCtx(ctx)` +
  `ExecContext`/`QueryRowContext`; `context.WithTimeout(…, 10*time.Second)` around `Close`;
  optionally a `sync.WaitGroup` drain.
- **Impact:** Bounded shutdown, no leaked goroutines, per-call timeouts during outages.

### M6. Owner `@refresh` is strictly sequential (plus minor N+1)

- **Where:** `refresh.go:39-45`; `ensureWarningMessage` re-queries `GetChannelByID` +
  `CountEventsByGuild` per guild (`handler_honeypot.go:174-178`)
- **What:** 1–4 REST calls per guild in series; Discord rate-limit buckets are per-channel so
  cross-guild calls parallelize cleanly. At scale the loop takes minutes instead of seconds.
  Also 2N queries where 2 would do.
- **Fix:** `errgroup.SetLimit(8)` around `ensureWarningMessage` with `atomic.Int64` tally;
  pass the already-loaded `Channel` in; one `GROUP BY guild_id` count query.
- **Impact:** ~8–10x faster refresh. Owner-only and rare, hence medium not high.

### M7. disgo's logger not wired

- **Where:** `bot.go:63` (client construction)
- **What:** The app builds a dedicated `LOG_LEVEL`-controlled logger, but disgo v0.19.6
  defaults to `slog.Default()` — gateway reconnects, resumes, rate-limit hits (the daemon's
  primary health signals) go to a differently-formatted stream at fixed Info level.
  `LOG_LEVEL=debug` never reaches the layer you'd actually debug.
- **Fix:** `dbot.WithLogger(log)` in the ConfigOpts (one line), or `slog.SetDefault(log)`.
- **Impact:** Unified log stream; gateway health observable and level-controllable.

### M8. Failure logs hide their causes

- **Where:** `handler_message.go:165` (alert fallback failure at Debug);
  `handler_honeypot.go:245` (root cause at Debug) vs. Warn call sites at `setup.go:69`,
  `refresh.go:43`, `handler_honeypot.go:199` that carry no `err`
- **What:** `sendAlert`'s own contract says alerts are "never silently dropped," yet total
  delivery failure logs at Debug — invisible at default level. Warning-message helpers return
  bare bools; operators see "refresh failed" with no reason and must redeploy at debug to
  find out.
- **Fix:** Alert failure → Error. Warning helpers return `error`; attach `"err", err` at the
  Warn sites; drop the Debug duplicates.
- **Impact:** Failures diagnosable from a single default-level log line.

### M9. `onMessageCreate` god function; orchestration untestable

- **Where:** `handler_message.go:16-131` (115 lines)
- **What:** One function does routing, dedup, reaction, exemption gathering, DM-before-ban
  race, ban/softban branching, unknown-ban classification, event recording, log assembly,
  and warning refresh. Tests cover only leaf predicates; the softban/unban/alert sequence —
  the code most likely to harbor subtle bugs (already needed the `JSONErrorCodeUnknownBan`
  carve-out) — has zero coverage because it's welded to `b.Client.Rest`.
- **Fix (proportional, no disgo mocks):** extract `gatherExemptionInputs(...)` and
  `moderate(cfg, guildID, channelID, msg)`; optionally lift the pure "what should happen"
  decision (action, shouldDM, shouldAlert) into `rules.go` for table tests.
- **Impact:** The riskiest logic becomes testable; single obvious insertion point for new rules.

### M10. `handler_honeypot.go` is a 348-line grab-bag with a misleading name

- **Where:** `internal/bot/handler_honeypot.go`
- **What:** Despite the name it does not handle honeypot triggers (that's
  `handler_message.go`). It holds five unrelated responsibilities: `/honeypot` command,
  modal construction, `validateConfig` (a pure rule), the entire warning-message lifecycle
  (~145 lines, called from four files), permission helpers, and `replyEphemeral`.
- **Fix (pure file moves):** `warning.go` (warning lifecycle); rename to
  `handler_config.go`; `validateConfig` → `rules.go`; helpers → `bot.go` or `helpers.go`.
- **Impact:** The second-most intricate subsystem becomes discoverable; no API change.

### M11. Runtime image ~6x larger than needed

- **Where:** `Dockerfile:10-16`
- **What:** debian-bookworm-slim + `apt-get install ca-certificates` + `useradd` for a fully
  static `CGO_ENABLED=0` binary. ~90 MB image, apt network fetch on cold builds, larger CVE
  surface — and builds happen on the prod host (M12).
- **Fix:** `gcr.io/distroless/static-debian12:nonroot` — ships CA certs and a nonroot user;
  drops the whole apt/useradd layer. ~10–15 MB.
- **Impact:** Smaller, faster, reproducible runtime layer.

### M12. Deploys build on the prod server; no registry, no rollback, weak SSH trust

- **Where:** `deploy-prod.yml:42`, `deploy-dev.yml:57` (`up -d --build --force-recreate`);
  `ssh-keyscan` at `deploy-prod.yml:35`, `deploy-dev.yml:48`; no healthcheck, no log
  rotation, no resource limits in either compose file
- **What:** Go compiles on the production box; no versioned artifact means rollback =
  rebuild an old SHA (impossible if the build breaks); dangling images accumulate;
  `--force-recreate` restarts even when nothing changed. `ssh-keyscan` per run re-learns the
  host key, so MITM protection is nil for a channel carrying `BOT_TOKEN`. A wedged gateway
  websocket never triggers `restart: unless-stopped`; default json-file logging is unbounded.
- **Fix:** Build/push SHA-tagged images to GHCR in CI (`cache-from/to: type=gha`); server does
  `compose pull && up -d`. Pin the server host key as a secret. Add heartbeat-file
  healthcheck + autoheal, `logging: {max-size: "10m", max-file: "3"}`, `mem_limit: 256m`.
- **Impact:** Instant rollbacks, no prod build load, restored SSH MITM protection,
  auto-recovery from silent gateway hangs, bounded disk.

## Low Severity

### L1. `.dockerignore` misses cache-busting files
`.dockerignore` — `.DS_Store` (exists at repo root, mutated silently by Finder) and
`.superpowers/` are sent in the build context and invalidate the `COPY . .` layer — and thus
the entire `go build` — when no source changed. Add them plus `Dockerfile`,
`docker-compose*.yml`, `.golangci.yml` (or invert to an allowlist).

### L2. `honeypot_events` grows unbounded
`queries.go:160-170`, schema `0001_init.sql:13-19` — every row is attacker-created; nothing
prunes. Only the aggregate count is ever read. Retention sweep (e.g. 90 days) or replace with
a per-guild counter column (pairs with M3).

### L3. DM channel cache not invalidated on send failure
`handler_message.go:188-191` — if `CreateMessage` fails with unknown-channel, the stale ID is
cached for 24h and every DM to that user fails. `isPermanentChannelError` already exists;
on permanent error, `b.DMs.Delete(userID)` and retry once.

### L4. Dedup TTL is dead weight
`handler_message.go:36-39` — `SetIfAbsent(key, …, 30*time.Second)` + `defer Delete` means the
30s never applies; the guard only covers handler lifetime (~0–2.3s). Either drop the defer and
make 30s the real re-punish window, or shrink the TTL to an honest value. Pick one mechanism.

### L5. Bare goroutines without panic recovery
`handler_message.go:42,70,82` — disgo recovers panics in listeners but not in self-spawned
goroutines; a panic kills the daemon for every guild. A `b.safeGo(fn)` wrapper with
`defer recover()` + Error log is cheap insurance.

### L6. `os.Exit(1)` skips deferred store close
`main.go:46,54` — restructure into `run() error`, exit once from `main`. Cosmetic (WAL
recovers), but the code intends a clean close.

### L7. Over-exported package-internal identifiers
`Normalize`, `Obfuscate`, `IsTriggerMessage`, `IsExempt`, `UnbanExpired`, all 8 template
functions, and `Bot`'s five fields (`Client`, `Store`, `Log`, `Dedup`, `DMs`) have zero
consumers outside `package bot` (tests are in-package). Lowercase them; keep `Bot`, `New`,
`Start`, `Close`.

### L8. Near-duplicate `onChannelDelete` / `onThreadDelete`
`housekeeping.go:7-38` — line-for-line identical except the ID field. Extract
`handleContainerDelete(guildID, containerID, kind)`.

### L9. Duplicated DM footer string
`templates.go:46,55` — `const dmFooter = …` used by both templates.

### L10. Admin-role map rebuilt per trigger
`handler_message.go:54-63` — iterates every guild role and allocates a map per honeypot hit.
Invert: for each of the member's few role IDs, check the role cache directly; no map needed.
Rate-limited path, so minor.

### L11. `Normalize`/`Obfuscate` double pass
`lookalike.go:41,53` — `strings.ToLower` copy + rebuild. Single pass with
`unicode.ToLower(r)` + `b.Grow(len(s))`. Cold path (setup only); do only if touching anyway.

### L12. Duplicate warning messages deleted one call at a time
`handler_honeypot.go:205-208` — per-message deletes share a tight per-channel bucket;
`BulkDeleteMessages` handles 2–100 in one call (with per-message fallback for >14-day-old).
Rare path, but sits inside the un-deferred interaction (H4).

### L13. Post-ban log + warning refresh run sequentially
`handler_message.go:128-129` — independent targets; an errgroup pair shaves one round-trip of
goroutine lifetime per moderation event. Marginal.

### L14. Check-then-act race on DM channel creation
`handler_message.go:181-189` — same user tripping honeypots in two guilds concurrently can
double-`CreateDMChannel`. Idempotent on Discord's side; `singleflight` if it ever matters.

### L15. `LOG_LEVEL` documented but not plumbed through compose
`.env.example:6` vs compose `environment:` blocks — add `LOG_LEVEL: ${LOG_LEVEL:-info}` to
both so ops can flip debug without a YAML edit.

### L16. Error logs missing correlation IDs
Various (`handler_message.go:117`, `housekeeping.go`, `handler_honeypot.go:70,202,222,345`) —
multi-guild daemon; add `guild`/`channel`/`user` attributes or `b.Log.With("guild", guildID)`.

### L17. Package doc comment on the wrong file
`lookalike.go:1-2` — move `// Package bot …` to `bot.go`.

## Deliberate design notes (do NOT "fix")

- `time.Sleep(250ms)` in the softban path (`handler_message.go:102`) and the 2s bounded DM
  wait (`:88-91`) are safe **because** `WithAsyncEventsEnabled()` (`bot.go:61`) gives each
  event its own goroutine. Worth a code comment in case async events is ever reconsidered.
- `_ = tx.Rollback()` after a failed Exec is idiomatic; not a swallowed error.
- `(nil, nil)` for not-found is a consistent store convention; the bug is the caller
  conflation (H2), not the convention.
- `store` importing `snowflake.ID` is fine — a value-type ID, not a client dependency.
- `interactionReplier` (`handler_honeypot.go:339-341`) is a textbook consumer-defined
  interface. Keep.
